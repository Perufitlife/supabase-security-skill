# Twitter / X thread (8 tweets)

**1/**
yesterday I scanned my own supabase project for the first time

found 17 tables anyone with the anon key from my js bundle could read, write, or delete

b2b_leads. engagement_emails. winback_queue. 😬

**2/**
how it happened: supabase's default has been to auto-grant CRUD on public-schema tables to the `anon` role

most projects rely on Row Level Security as the safety net. on 17 of mine, I'd forgotten to enable it

**3/**
they're changing this default
→ may 30, 2026 — new projects
→ oct 30, 2026 — enforced on all existing projects

but that won't fix what's already there. existing grants stay.

so I had to actually check, today, what's exposed

**4/**
wrote a 250-line node script. queries pg_class, pg_policies, pg_proc, pg_default_acl. checks storage buckets and auth config

outputs a self-contained html report with copy-paste fix sql on every finding

no deps. no saas. your token stays on your machine

**5/**
results on 2 projects:

internal crm (auth-only): 0 critical, 11 high (mostly intentional security definer APIs)

public web app: SEVENTEEN critical. all rls-disabled tables with anon CRUD

fixed in one transaction generated by the tool

**6/**
why not use existing scanners? supaexplorer, audityourapp, vibe app scanner — they work

but they all want me to send my project ref + creds to their servers just to find out if my doors are locked. weird trade for a security tool

**7/**
tradeoff is real: SaaS scanners have nicer features (subdomain discovery, team collab, PDFs)

local script gives you: source code you can read, no recurring cost, CI-friendly, zero data sharing

pick the one that matches your trust model
